# UI 最適化ワークフロー

このファイルは UI 最適化ワークフロー全体の **単一情報源 (SSoT)** である。`ui-optimizer` agent は権限境界のみを定義し、評価基準・コスト規律・暴走対策・出力形式はすべて本ファイルを参照する。

実装完了後の UI を Playwright MCP で開き、視覚検査と最適化を行う。TDD ワークフロー (`tdd-workflow` skill) の REFACTOR フェーズ完了後に任意で起動する位置付け。

## 前提条件

以下が揃っていない場合は、ユーザーに確認してから進めること。勝手に進めない。

- Playwright MCP がプロジェクトに登録済み（`claude mcp list` に `playwright` がある）
- dev サーバが起動済み（別ターミナルで `bun run dev`）
- プロジェクト `CLAUDE.md` に `## UI Standards` セクションがある

## Phase 1: 計画

1. プロジェクト `CLAUDE.md` の `## UI Standards` セクションを読む
2. 検査対象の URL とコンポーネントファイルを特定する
3. 計画（対象 URL・対象ファイル・想定される検査観点）をユーザーに提示
4. ユーザーのレビューゲートを挟むこと。勝手に Phase 2 に進まない

## Phase 2: 視覚検査と最適化

### サブエージェント

`.claude/agents/ui-optimizer.md`（`subagent_type: "ui-optimizer"`）

### 手順

1. Phase 1 で確定した検査対象 URL とファイルパスを引き継ぐ
2. 検査対象 URL に curl 等で疎通確認。応答しない場合はユーザーに `bun run dev` の起動を依頼して停止
3. 検査対象ファイルを Read してコンポーネント構造を把握
4. `mcp__playwright__browser_navigate` で対象 URL を開く
5. `mcp__playwright__browser_snapshot` で DOM 構造取得（テキスト、軽量）
6. 3 viewport で `mcp__playwright__browser_take_screenshot`
   - 375px (mobile) / 768px (tablet) / 1280px (desktop)
7. 評価基準と突き合わせ NG 項目を以下のフォーマットで列挙:
   ```
   - [重要度: 高/中/低] [項目] 現状: ... → 期待: ...
   ```
8. 最適化ループ（**最大 3 イテレーション**で打ち切る）
   1. 重要度の高い NG 項目から 1 件を選ぶ
   2. 該当ファイルを Edit
   3. ページをリロード（`browser_navigate` 再実行）
   4. `browser_snapshot` で差分確認
   5. 改善されていれば次の項目へ
   6. 改善されなければ Edit を rollback してユーザーに報告
9. 収束判定（以下のいずれかで停止）:
   - 直前の DOM スナップショットと一致（変更なし）
   - イテレーション上限 3 回到達
   - 致命的な NG が残存 → ユーザーに判断を仰ぐ
10. 完了後、ユーザーのレビューゲートを挟むこと

### htmx 観点での留意点

- 部分更新がある画面では、トリガー操作後の DOM スナップショットを再取得してから評価する（初期 GET の DOM だけで判断しない）
- フラグメントを返すエンドポイントは「0 件のとき」の表示が崩れないか確認する

## Phase 3: 適用判定

1. before/after サマリと変更ファイル一覧をユーザーに提示
2. 適用するかの最終承認をユーザーに求める
3. 承認後、関連テストが通ることを `bun test` で確認
4. テストが落ちた場合は変更を rollback してユーザーに報告

## 汎用評価基準

1. **レイアウト**: 横スクロール発生・要素はみ出しがない
2. **コントラスト**: WCAG AA (4.5:1) 以上
3. **タップ領域**: ボタン・リンクが概ね 44px 四方以上
4. **レスポンシブ**: 3 viewport いずれでも破綻しない
5. **視覚的階層**: 見出しサイズ・余白・整列が一貫している
6. **状態表現**: ローディング・空状態・エラー状態が考慮されている

## このプロジェクト固有の留意点

- **pico.css v2 のセマンティクス優先**: ユーティリティクラスを足すのではなく、`<main class="container">`・`<form role="group">` 等の semantic HTML + pico デフォルトを活かす
- **Hono JSX**: `react` から import しない。`hono/jsx` の `FC` を使う
- **htmx**: 完全ページ vs フラグメント返却の使い分けを壊さない
- **ランタイムは Bun**: テスト・依存解決は Bun 前提

## コスト規律

- スクショは「初回・最終・viewport 切替時」のみ
- 中間チェックは `browser_snapshot`（テキスト DOM）を優先
- 1 セッション内のスクショ枚数を 12 枚以内に抑える

## 暴走対策

- イテレーション上限: **3 回**（厳守）
- バックエンドロジック・schema (`src/schema.ts`)・テストファイル・マイグレーション (`drizzle/`) は絶対に変更しない

## 制約

- 自動でループを継続しない（イテレーション上限を厳守）
- スクショは必要最小限。中間チェックは `browser_snapshot` を優先
- バックエンドロジック・schema・テストファイルは変更しない
- 各 Phase でレビューゲートを挟み、勝手に次に進まないこと

## 最終報告の出力形式

最終報告は以下の構造でユーザーに提示する:

1. **検査サマリ**: viewport ごとの発見事項
2. **適用差分**: 変更したファイルとその要約
3. **before / after**: 主要 viewport のスクショ参照
4. **未対応項目**: 自動適用しなかった NG とその理由
5. **テスト結果**: `bun test` のパス/フェイル
6. **プロセスの終了**: `bun dev` で起動しているプロセスを必ず終了し、port 3000 を必ず開けて終了させる
7. **スクリーンショットの削除**: 作業中に使ったスクショは最後に全て削除する

ユーザーが最終承認するまで PR 作成・コミットはしない。
