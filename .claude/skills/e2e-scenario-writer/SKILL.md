# E2E Phase 1: シナリオ列挙

`e2e-test-workflow` skill の **Phase 1** に対応する詳細ルール集。`e2e-scenario-writer` agent はこの skill を **必ず READ** して従う。

ルートの `CLAUDE.md`（`## E2E テスト規約` 含む）と `.claude/rules/testing/test-philosophy.md` のテスト思想に必ず従うこと。

## 手順

1. 対象機能（ハンドラ・ビュー・フォーム）の **現状の挙動** を把握する。**実装ファイルを読み、必要に応じて Playwright MCP（`mcp__playwright__browser_navigate` / `browser_snapshot`）でローカルブラウザを観察し、ブラウザ上で何が起きるかを確定させる**（後追いシナリオなので TDD Phase 1 と異なり実装参照を推奨）
2. ハッピーパス **2-3 件**、異常系 **1-2 件** を厳選する。テストピラミッドで E2E は最少
3. `tests/e2e/<feature>.spec.ts` に `test.describe` + `test.skip()` でシナリオを列挙する（コードは書かない）
4. 列挙完了後、必ずユーザーのレビューゲートを挟む。勝手に Phase 2 へ進まない

## ルール

- 単体・結合（`src/**/*.test.ts`）でカバー済の検証は重複させない（バリデーションメッセージの個別文言など）
- E2E は **画面遷移・htmx の動的差し替え・ブラウザ駆動でしか壊れない振る舞い** に絞る
- 「こうあるべき」ではなく **「こうなる」** をシナリオ名にする。実装が直感に反していても E2E は仕様批判の場ではない（仕様の正しさは TDD で担保済みの前提。直したい場合は `tdd-workflow` に戻る）
- ケース名は日本語で「〜のとき〜であること」と書き、生きたドキュメントにする
- 実 DB 状態を前提にする場合は `test.beforeEach` の `truncateAll()` で初期化されることを念頭に置く

## ケース選定ガイド（このプロジェクトの場合）

- **ハッピーパス候補**: フォーム送信→`#messages` フラグメント差し替え／テーマ切り替え→`HX-Refresh` 後の `data-theme` 反映
- **異常系候補**: 必須フィールド未入力時の `aria-invalid` 表示／エラー時のフラグメント差し替え先（`#message-form`）の整合
- **書かない例**: バリデーションメッセージ個別文言（単体テストでカバー済）／DB 50 件上限（結合テストでカバー済）

## 出力形式

- 対象 spec ファイルのパス
- 列挙したシナリオの一覧（`test.skip()` のラベル）
- 各シナリオが検証する振る舞いの説明（1-2 文）
- 単体・結合との重複がないことの根拠（参照した既存テストファイル）
